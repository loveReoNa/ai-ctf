```
封面
摘要（半页）
目录

1. Introduction（引言）
    1.1 CTF 竞赛中的工具生态现状
        - nmap / sqlmap / OWASP ZAP / Metasploit / hashcat 等工具的角色
    1.2 LLM 与传统安全工具的结合困境
        - 工具输出格式不统一、噪音多、上下文窗口浪费
        - 从「人类可读」到「LLM 可解析」的转换需求
    1.3 项目目标与我的角色定位
        - 作为「手」为 Agent 大脑提供干净、结构化的工具能力
    1.4 报告结构说明

2. Project Overview（项目概述）
    2.1 系统架构中的模块二位置
        - 工具层是连接 Agent 大脑与靶机环境的桥梁
        - 输入来自 A（Agent 的工具调用请求），输出供给 C（日志系统）
    2.2 安全工具生态全景
        - 侦察/扫描/利用/破解/后渗透五大类工具的选型论证
    2.3 团队分工与协作模式
        - 我与 A 的接口定义（Schema 对齐）
        - 我与 C 的日志对接

3. Personal Technical Contributions（个人技术贡献）— 核心章节
    3.1 通用工具封装器（ToolWrapper）设计
        3.1.1 统一接口规范
            - 输入：tool_name + parameters（dict）
            - 输出：标准化的 ToolResult（stdout + stderr + exit_code + parsed_data）
            - 子进程管理：超时控制、环境隔离、资源限制
        3.1.2 多工具适配策略
            - 命令行工具（nmap, sqlmap, hashcat）：subprocess 封装
            - Python SDK 工具（OWASP ZAP API）：直接调用封装
            - Docker 化工具：容器内执行 + 结果提取
        3.1.3 执行安全约束
            - 白名单命令校验
            - 网络访问范围限制
            - 防止命令注入的输入清洗
```

## 写作注意事项（全组通用）

1. **第三章是最长章节**，占全文 40-50%，你的模块写 8-10 页，其他模块用 2-3 页概述 + "详见组员 X 的报告"
2. **证据驱动**：每个技术点引用文件路径 + 代码行数 + 测试结果
3. **参考文献 APA 第7版**，每人 ≥10 篇，可共用核心文献（ReAct, MCP, LiveMCPBench）但结合自身模块追加专属文献
4. **避免重复**：四人共享第 1/2 章框架但**各写各的语言**，第 3 章完全独立，第 7 章从各自专业角度反思
5. **相互引用**：可以用 "Refer to Member X's report for detailed implementation of Module Y" 减少重复
